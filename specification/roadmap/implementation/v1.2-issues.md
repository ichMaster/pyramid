# v1.2 — GitHub Issues

Issues for phase **v1.2 — TTS output (spoken replies)** (version **v1 — Voice**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v1.2). One phase per file; PYR IDs continue from v1.1 (PYR-004…007). Each issue ships with the tests that encode its DoD (see [ARCHITECTURE.md](../../ARCHITECTURE.md) §Testing and CI).

v1.2 proves the **output half** of the voice loop before any ASR exists: the device takes the LLM `reply` from the v0 loop, sends it to a cloud **Ukrainian TTS**, and plays the returned audio through the Echo Base speaker (the v1.1 playback path). Input stays text-over-serial; only the output becomes voice. Build order across v1 is deliberately output-first, so this lands before v1.3 (ASR).

## v1.2 — TTS output (spoken replies)

### PYR-008 — Cloud TTS client (Ukrainian → PCM16)

**Description:**
Add a client that turns text into speech: an HTTPS call to a cloud Ukrainian TTS that takes `reply.text` and returns audio the device can play. To avoid an on-device audio decoder (MP3/Opus are heavy on the ESP32), target **raw PCM16** output at the playback sample rate (`AUDIO_SAMPLE_RATE`, 16 kHz).

**Provider (decided): ElevenLabs**, `output_format=pcm_16000` — raw 16 kHz mono PCM16, matching `AUDIO_SAMPLE_RATE` (no on-device decoder), Ukrainian via `eleven_multilingual_v2`, `xi-api-key` auth. Endpoint: `POST https://api.elevenlabs.io/v1/text-to-speech/{voice_id}?output_format=pcm_16000`, body `{"text": <reply>, "model_id": "eleven_multilingual_v2"}`. (Google `LINEAR16` / OpenAI `pcm` remain documented fallbacks if needed.)

**What needs to be done:**
- Add a TTS config block to `config.h`: `TTS_API_KEY` (`xi-api-key`), `TTS_VOICE_ID`, `TTS_MODEL` (`eleven_multilingual_v2`), `TTS_ENDPOINT` base, and `TTS_SAMPLE_RATE` = `AUDIO_SAMPLE_RATE` (16 kHz). Behavior is config-driven (config is the source of truth); in v2+ voice params move into the `Role`.
- Implement the HTTPS POST (reuse `WiFiClientSecure` + `HTTPClient`): set `xi-api-key` + `content-type: application/json` + `accept: audio/pcm`, build the JSON body from `reply.text`, send, and read the **raw PCM16** response straight into a buffer (no decode).
- Keep the request building (JSON, with quote/UTF-8 escaping) in a **pure, host-testable** helper — a `tts_api.h` mirroring `chat_api.h`. (ElevenLabs returns raw audio, so there's no response JSON to parse; if a fallback JSON+base64 provider is ever used, its base64-decode helper goes here too.)

**Dependencies:** None (builds on the existing HTTPS stack and v1.1 audio buffers).

**Expected result:**
Given a Ukrainian text string, the client returns its speech as a PCM16 buffer at the playback sample rate.

**Acceptance criteria:**
- [ ] Direct HTTPS (TLS) call to the cloud TTS returns audio for a given text
- [ ] Request is built from the text plus voice/format/endpoint/key in `config.h` (UTF-8 Ukrainian preserved)
- [ ] The response is delivered as PCM16 at `AUDIO_SAMPLE_RATE` (raw, or decoded from the provider's format)
- [ ] Request building (and any base64/JSON extraction) is unit-tested against a recorded/mock response (`pio test -e native`)
- [ ] On-device: a known phrase fetches non-empty audio (manual check; logged length/bytes)

### PYR-009 — TTS → playback pipeline (serial → LLM → TTS → speaker)

**Description:**
Wire the full output path: a typed `text_in` over serial goes to the LLM (existing v0 loop), the reply is sent to the TTS client (PYR-008), and the returned PCM16 is played through the Echo Base speaker using the v1.1 playback path. This realizes the v1.2 DoD — a typed prompt produces a *spoken* Ukrainian reply.

**What needs to be done:**
- After the LLM reply completes (it still streams to serial for debugging), pass `reply.text` to the TTS client and play the resulting PCM16 through `M5.Speaker` (reuse the v1.1 mic↔speaker switch so the bus is handed over safely).
- Keep `text_in` / `reply` mirrored to serial; drive LCD states through the turn (e.g. `thinking` → `speaking` → `idle`).
- Decide buffering: buffer-then-play is fine for v1.2 (fetch the full audio, then `playRaw`); chunked/streamed playback for lower latency is optional and may move to v1.4.
- Keep the persona/LLM config path unchanged; only the output is now voiced.

**Dependencies:** PYR-008 (and the v1.1 playback path, PYR-007 — done)

**Expected result:**
You type a line in the serial monitor and hear the assistant speak the reply in Ukrainian from the device.

**Acceptance criteria:**
- [ ] A typed serial line produces a spoken reply: `text_in` → LLM → TTS → speaker, end to end
- [ ] `text_in` and the LLM `reply` are still printed to serial for debugging
- [ ] The speaker plays the TTS PCM16 at the correct rate; the device returns to idle (mic mode) cleanly
- [ ] LCD reflects the turn (thinking / speaking / idle)
- [ ] Reuses the v1.1 I2S playback + mic/speaker switch (no second audio path); verified on hardware

### PYR-010 — TTS robustness (timeout / fallback / max length)

**Description:**
Harden the spoken-reply path for normal use: bounded TTS timeouts with a graceful fallback, a cap on how much text is synthesized, and clean handling of empty/failed audio — so a TTS hiccup never hangs or crashes the loop.

**What needs to be done:**
- Add a bounded TTS timeout (per-stage budget, like the LLM call); on TTS error/timeout, fall back to the **logged text reply** (it is already on serial) so the turn still completes — a "spoken-or-logged" fallback.
- Respect a **maximum reply length** (`TTS_MAX_CHARS`): truncate the text sent to TTS (or skip TTS) for over-long replies, and log when truncated.
- Handle empty / zero-length audio gracefully (skip playback, log, continue).
- Keep the error-mapping and length/truncation logic in pure helpers where feasible, host-tested.

**Dependencies:** PYR-009

**Expected result:**
The spoken-reply loop holds up: TTS timeouts/errors degrade to a logged reply, over-long replies are bounded, and the loop never hangs.

**Acceptance criteria:**
- [ ] TTS timeout/error is handled with a clear log and a fallback to the text reply — the loop never hangs or crashes
- [ ] Replies over `TTS_MAX_CHARS` are truncated (or TTS skipped) with a log line
- [ ] Empty/failed audio is handled gracefully; the device returns to idle
- [ ] Length/truncation and error-mapping logic are unit-tested (`pio test -e native`); a reliability pass over typical scenarios succeeds on hardware

## v1.2 scope notes

**Total effort:** ~1 week for a single developer (a short chain; PYR-008 is the bulk).

**Critical path:** PYR-008 → PYR-009 → PYR-010. The DoD (a typed prompt → spoken Ukrainian reply) is met at PYR-009; PYR-010 makes it robust.

**Phase DoD (ROADMAP §v1.2):** a typed prompt produces a spoken Ukrainian reply from the device.

**Provider (decided):** **ElevenLabs** with `output_format=pcm_16000` — raw 16 kHz PCM16, no on-device decoder, exact match to the playback rate, Ukrainian via `eleven_multilingual_v2`. The remaining open detail is the specific `TTS_VOICE_ID` (pick a voice in the ElevenLabs library at execution time). Google `LINEAR16` / OpenAI `pcm` stay documented as fallbacks only.

**Reused infrastructure:** the HTTPS/TLS client and config pattern from v0.2, and the I2S playback + mic↔speaker switch from v1.1 (PYR-007). No new audio path is introduced — TTS audio plays through the same `M5.Speaker` route.

**Secrets:** the TTS key joins the Wi-Fi creds + LLM key in the gitignored `config.h`; only `config.example.h` is committed (ARCHITECTURE §Security — extractable on-device keys are acceptable only under the private allowlist model).
