# v2.1 — GitHub Issues

Issues for phase **v2.1 — Server proxy** (version **v2 — Server platform**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v2.1). One phase per file; PYR IDs continue from v1.4 (PYR-014…017). Each issue ships with the tests that encode its DoD (see [ARCHITECTURE.md](../../ARCHITECTURE.md) §Testing and CI).

v2.1 is the **first v2 phase and the first server-side code in the project.** It moves the ASR→LLM→TTS turn off the device and onto our own backend: a FastAPI + websockets server terminates a single duplex WSS channel and runs the turn orchestrator, while the firmware becomes a thin WSS client that only streams audio/text and renders results. The serial path keeps working as a local debug client *through* the server. Two latency carry-overs from the v1.3 analysis land here — **sentence-streaming TTS** (rec #3) and **early/streaming playback** (rec #4) — plus streaming ASR partials (rec #1); together they cut time-to-first-audio versus v1's whole-reply playback. No `Role`/Canon, accounts, allowlist, emotion face, or deployment yet — those are v2.2–v2.6.

- Add the concrete server build/run/test commands to `CLAUDE.md` (per its note: when the first server code lands, document build/run/test there).

**Dependencies:** None (first `/server` code).

**Expected result:**
`uvicorn` serves a WSS endpoint; a client connects, sends `ping`, and receives `pong`; CI runs pytest green.

### PYR-019 — WS device↔server contract: codec/router + `hello` + `error.code` set

**Description:**
The wire contract between device and server — the integration seam that must not drift. Implement the full message set (JSON text frames + binary audio frames), the `hello` handshake with `proto_ver` negotiation, and the enumerated `error{code,msg}` set, all pinned by **contract tests** so device and server cannot diverge.

**What needs to be done:**
- Typed encode/decode for every message in ARCHITECTURE §WS device↔server: device→server `hello{device_token, proto_ver, audio_fmt}`, `listen_start`, `audio`(binary), `listen_stop`, `text_in{text}`, `ping`; server→device `asr_partial{text}`, `asr{text}`, `reply{text}` (delta-capable), `text_out{text}`, `tts_audio`(binary), `tts_end`, `error{code,msg}`, `config_updated`, `restart`, `pong`.
- Binary frames (`audio`, `tts_audio`) carry raw **PCM16 16 kHz mono**; text frames are JSON with a `type` field. One codec, one router that dispatches inbound messages and emits outbound ones.
- `hello` handshake: negotiate `proto_ver` (major.minor); reject an unknown **major** with `error{code:"proto_unsupported"}` and close (ARCHITECTURE §WS / §Cross-cutting).
- Define the enumerated `error.code` constants in one module (`wifi_lost`, `server_unreachable`, `proto_unsupported`, `unauthorized`, `rate_limited`, `asr_failed`, `llm_timeout`, `llm_failed`, `tts_failed`, `internal`) — ARCHITECTURE §Error handling.
- Idempotency rules: `listen_start`/`listen_stop` received outside the expected state are ignored; `restart` always returns the device to `boot` (ARCHITECTURE §Error handling).
- Provide a **fake-device** WS test client that speaks this protocol (the serial bridge doubles as this harness, ARCHITECTURE §Testing) for contract + later integration tests.
- Scope note: validating `device_token` (auth/allowlist) lands in v2.3 and the `config_updated` Role payload in v2.2 — here `hello` is accepted *structurally*; `emotion` frames arrive with the face in v2.4.

**Dependencies:** PYR-018.

**Expected result:**
Device and server exchange every message type through a single codec/router; the shapes are pinned by contract tests; proto negotiation rejects unknown majors.

**Acceptance criteria:**
- [ ] Encode/decode + router cover the full device↔server message set (text JSON + binary audio), matching ARCHITECTURE §WS exactly
- [ ] Contract tests pin each message shape (both directions); changing a contract must change its test
- [ ] `hello` negotiates `proto_ver`; unknown major → `error{proto_unsupported}` + close
- [ ] Enumerated `error.code` set lives in one module and is asserted by a contract test
- [ ] A fake-device WS client drives the protocol in tests (no hardware)

Port the v1 providers (Deepgram ASR, Anthropic streaming LLM, ElevenLabs TTS) into server-side adapters behind interfaces, and add a turn orchestrator that runs the pipeline and **streams each stage** out over the PYR-019 contract. Mock adapters keep CI offline and deterministic.

**What needs to be done:**
- Provider adapter interfaces (ASR / LLM / TTS) with **real** implementations — Deepgram (µ-law upload), Anthropic Messages **SSE streaming**, ElevenLabs `pcm_16000` — the same providers v1 used, now server-side with keys from `.env`.
- Turn orchestrator. Voice path: on `listen_stop`, run ASR → emit `asr_partial*` interims + final `asr` (streaming ASR, rec #1); feed transcript + rolling history to the LLM → stream `reply` deltas; synthesize TTS → emit `tts_audio` frames → `tts_end`. - Per-`Session` rolling history (`Message{role, text, ts}`), trimmed to a window / token budget before each LLM call (ARCHITECTURE §Sessions and history); audio frames are **never persisted**.
- **Mock LLM/ASR/TTS adapters** returning canned output so the pipeline runs deterministically and offline — the test seam ARCHITECTURE §Testing mandates (external AI is never called in CI).
- Prompt assembly behind a seam: in v2.1 the persona is still derived from firmware-era config; the configurable `Role` (Name + Canon) replaces it in v2.2, so keep assembly swappable.

**Dependencies:** PYR-019 (emits/receives via the contract + router).

**Expected result:**
A full turn runs server-side over fakes — `text_in → reply` and `audio → tts_end` — streaming each stage; history windows correctly; CI stays offline.

**Acceptance criteria:**
- [ ] ASR/LLM/TTS adapters behind interfaces, with real (Deepgram/Anthropic/ElevenLabs) **and** mock implementations
- [ ] Orchestrator runs ASR→LLM→TTS, streaming `asr_partial*`/`asr`, `reply` deltas, and `tts_audio`/`tts_end`
- [ ] Integration test (fake device + mocks): `text_in → reply` and `audio → tts_end`, asserting stage order and streaming
- [ ] Per-session rolling history windowed before each LLM call; audio not persisted (unit-tested)
- [ ] CI runs the pipeline with mocks only — no external AI calls

### PYR-021 — Sentence-streaming TTS (rec #3)

**Description:**
Carry over recommendation **#3** from the v1.3 latency analysis, now server-side. Instead of waiting for the whole reply, buffer LLM token deltas to **clause/sentence boundaries** and hand each completed phrase to TTS as it forms — so the first `tts_audio` chunk leaves the server while the model is still generating. This overlaps the TTS stage (~20 % of post-speech latency in v1.3) with LLM generation. It lives on the server because the v1.2 device-side attempt was choppy under single-thread TLS + audio contention.

**What needs to be done:**
- A pure, unit-testable **clause/sentence boundary splitter** that consumes a token-delta stream and yields complete phrases at sentence/clause punctuation — handling Ukrainian punctuation, common abbreviations (no false split), and a **max-length flush** so one long clause doesn't stall the stream.
- Wire it into the orchestrator: each completed phrase is synthesized and streamed as `tts_audio` immediately, before the LLM stream finishes; `reply` deltas still stream to the device in parallel (for the on-screen transcript).
- Keep the buffered whole-reply path as a fallback for very short replies.

**Dependencies:** PYR-020.

**Expected result:**
The first `tts_audio` frame is sent before the LLM finishes generating; speech begins earlier with no audible seams between phrases.

**Acceptance criteria:**
- [ ] Boundary splitter is pure + unit-tested (sentence end, clause end, abbreviation no-split, max-length flush, Ukrainian punctuation)
- [ ] Integration test: first `tts_audio` frame is emitted before the mock LLM stream completes (phrase-by-phrase)
- [ ] No regression in the full `audio → tts_end` integration test
- [ ] Measured time-to-first-`tts_audio` drops vs. the whole-reply baseline (numbers recorded)

### PYR-022 — Per-stage timeouts + enumerated `error.code` mapping

**Description:**
Make the server turn **bounded** and its failures **legible** — implement the per-stage timeout budgets and map every provider/connection failure onto the enumerated `error{code}` set, returning the turn to idle on breach without dropping the session.

**What needs to be done:**
- Per-stage timeout budgets (ARCHITECTURE §Error handling): **ASR ≤ 5 s, LLM first token ≤ 8 s, TTS first audio ≤ 3 s**. On breach, abort the stage, emit `error{code}`, end the turn → idle.
- Map provider/connection failures to the enumerated codes: `asr_failed`, `llm_timeout`, `llm_failed`, `tts_failed`, `rate_limited`, `server_unreachable`, `internal` (`unauthorized` arrives with auth in v2.3; `proto_unsupported` is handled in PYR-019).
- Degrade rather than hang: an interrupted turn is **abandoned, not queued**; the session stays connected and ready for the next turn (ARCHITECTURE §Error handling, §Sessions).

**Dependencies:** PYR-020 (wraps the orchestrator stages), PYR-019 (the `error` contract + code set).

**Expected result:**
Any stalled or failed stage produces the correct enumerated `error{code}` and a clean return to idle; the socket/session stays usable.

**Acceptance criteria:**
- [ ] ASR/LLM/TTS each bounded by its budget; breach → `error{code}` + turn ends → idle
- [ ] Provider failures mapped to the enumerated codes (unit-tested mapping)
- [ ] Integration tests over stalling/failing mocks assert the right `error.code` per stage
- [ ] An aborted turn leaves the session connected and ready for the next turn

### PYR-023 — Firmware WSS client + early/streaming playback (rec #4)

**Description:**
Convert the firmware from a direct-to-cloud client into a thin **WSS client** of our server. The device streams mic audio and typed text up, plays returned `tts_audio` **chunk-by-chunk as it arrives** (early/streaming playback, rec **#4**), and renders state — all intelligence now lives server-side. Serial becomes a local debug client bridging `text_in`.

**What needs to be done:**
- Open one duplex **WSS** channel to the server; send `hello{device_token, proto_ver, audio_fmt}`, then `listen_start`/`audio`(bin)/`listen_stop` for voice and `text_in` for typed/serial input; handle `asr`/`asr_partial`/`reply`/`text_out`/`tts_audio`/`tts_end`/`error`/`restart`/`pong`.
- **Early/streaming playback (rec #4):** play each `tts_audio` frame as it arrives (server-paced) instead of buffering the whole reply — audio starts on the first chunk. The server pacing sidesteps the on-device underrun that forced buffered playback in v1.2.
- **Remove the on-device cloud ASR/LLM/TTS HTTP path** (the v1 `cloud.*` calls): the device no longer holds those API keys; config keeps only Wi-Fi + server URL + `device_token` (intelligence off-device — MISSION principle).
- Serial bridge: a typed serial line becomes `text_in` over the WS; server replies (`text_out`/`reply`, `tts_audio`) still play — serial is the local debug client (ARCHITECTURE §Testing — the serial bridge doubles as the fake-device harness).
- WSS reconnect with the v0.3 exponential backoff (0.5→8 s, jittered); on Wi-Fi/server loss show `offline`, abandon the in-flight turn, resume `idle` on reconnect (ARCHITECTURE §Error handling); `restart` → boot. Keep the v1.4 turn-state machine driving the LCD.
- Host-test the WS frame codec / serial line bridge on the native env (`pio test -e native`).

**Dependencies:** PYR-019 (the contract the firmware speaks — required to build), PYR-020 (a working server turn — required for end-to-end hardware verification). Benefits from PYR-021/PYR-022 for the full first-audio + error experience.

**Expected result:**
The device runs the full voice loop **through our server** with the same UX as v1 and **lower time-to-first-audio** — playback starts on the first streamed `tts_audio` chunk.

**Acceptance criteria:**
- [ ] Firmware is a WSS client speaking the PYR-019 contract (hello / listen / audio / text_in ↔ asr / reply / tts_audio / tts_end / error / restart)
- [ ] `tts_audio` plays chunk-by-chunk as it arrives (first-chunk playback), not buffered whole
- [ ] On-device cloud ASR/LLM/TTS removed; config holds only Wi-Fi + server URL + `device_token`
- [ ] Serial typed input bridges to `text_in` and the reply plays back (debug client works)
- [ ] WSS reconnect/backoff + `offline`/`restart` handling; WS codec host-tested (`pio test -e native`)
- [ ] Verified on hardware end-to-end through the server: same voice UX as v1, first-audio starts before the full reply is synthesized (vs. the v1 baseline)

## v2.1 scope notes

**Total effort:** ~3–4 weeks for a single developer — this phase stands up the entire `/server` *and* rewrites the firmware's transport. PYR-018 → PYR-019 is the spine; after it the server and firmware tracks run in parallel.

**Critical path:** PYR-018 → PYR-019 → PYR-020 → PYR-023. The "end-to-end through our server" half of the DoD is met at PYR-023; the "lower time-to-first-audio" half is delivered by PYR-021 (sentence-streaming TTS) + PYR-020's streaming ASR partials (#1) + PYR-023's early playback (#4).

**Phase DoD (ROADMAP §v2.1):** the device runs end-to-end through our server, with the same voice experience as v1 and a **lower time-to-first-audio** — playback starts on the first streamed TTS chunk rather than after the full reply is synthesized.

**Latency carry-overs from v1.3:** rec **#1** (streaming ASR partials) and the per-stage streaming live in PYR-020; rec **#3** (sentence-streaming TTS) is PYR-021; rec **#4** (early/streaming playback) is PYR-023. The device-side pre-warm tried and reverted in v1.4 is **not** revisited — pacing now lives server-side.

**Explicitly out of scope (later v2 phases):** the configurable `Role` (Name + Canon) and `config_updated` payload — **v2.2**; accounts, activation by code, allowlist, and `unauthorized` enforcement — **v2.3**; the on-screen **emoji face** + `emotion` frame — **v2.4**; public deployment (Fly.io) + CI/CD + security tests — **v2.5/v2.6**; active-listening mode — **v2.7**; multi-session hub (many concurrent clients with shared resources) — **v4.3**. v2.1 terminates a **single** duplex channel.

**Contracts pinned here:** the WS device↔server message set and the enumerated `error.code` set (ARCHITECTURE §WS device↔server, §Error handling). Any later change to these updates ARCHITECTURE.md **and** its contract test in the same commit (CLAUDE.md / ARCHITECTURE §Testing).

**Reused infrastructure:** the v1 providers (Deepgram/Anthropic/ElevenLabs) ported into server adapters; the v1.4 turn-state machine (`states.h`) + VAD (`vad.h`) and the v0.3 Wi-Fi supervisor + `backoff.h` (reused for WSS reconnect) on the firmware side. New: the entire `/server` package, `/tests` (contract + integration over a fake device + mock LLM/ASR/TTS), and `.github/workflows/ci.yml`.
