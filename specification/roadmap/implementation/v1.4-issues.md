## v1.4 — States and UX

### PYR-014 — Turn-state machine + LCD states

**Description:**
Today the loop sets ad-hoc LCD strings (`showStatus("listening")`, `"thinking"`, `"speaking"`, `"error"`, `"offline"`, …) scattered across `recordWhileHeld`/`handleTurn`/`voiceTurn`/`serviceWiFi`. Replace that with a single **turn-state machine** so the displayed state is always correct and there is one place every path updates. This is the foundation the rest of v1.4 hooks into.

**What needs to be done:**
- Define a pure, host-testable `TurnState` enum (`Idle`, `Listening`, `Thinking`, `Replying`, `Error`, `Offline`) and a transition function in a new header (e.g. `states.h`): `nextState(current, event)` plus a `label(state)` (and ideally a color) for rendering.
- Centralize LCD rendering through the state machine on the 128×128 LCD — label + a glance-readable color — and route every existing transition (record → ASR → LLM → TTS → playback, plus Wi-Fi loss/reconnect) through it.
- Keep **Replying** (speaking) distinct from **Thinking** (waiting) so the user can tell "still working" from "talking back".
- Remove the ad-hoc `showStatus` string call sites in favor of the state machine.

**Dependencies:** None (a refactor of the existing `showStatus` calls + a new pure header).

**Expected result:**
The LCD always shows the true current state, driven by one state machine; transitions are unit-tested.

**Acceptance criteria:**
- [ ] `TurnState` enum + transition function live in a pure header, host-tested (`pio test -e native`)
- [ ] LCD shows listening / thinking / replying / error / idle / offline consistently and legibly
- [ ] All transitions route through the state machine (no stray ad-hoc `showStatus` strings remain)
- [ ] Verified on hardware: the state is correct and readable through a full turn

### PYR-015 — Pause-based end-of-utterance (VAD)

**Description:**
Today end-of-utterance is **button release** (push-to-talk). Add silence-based endpointing so the user can speak without timing the button: detect the end of speech by a **pause** (silence threshold sustained for a hangover window), bounded by **`recog_patience`**. Push-to-talk stays as an explicit override; the two modes must coexist naturally.

**What needs to be done:**
- Add a pure endpointing helper (e.g. `vad.h`, or extend `audio.h`): given per-chunk level (peak/RMS), decide end-of-utterance when silence is sustained for a hangover window after speech was seen; cap total listening by `recog_patience` and the hard `REC_MAX_MS`. Host-test with synthetic frame sequences (speech→silence→end; never-ends→patience cap; all-quiet→no false trigger; speech-with-short-gaps→no premature cut).
- Add config knobs to `config.h`: `VAD_SILENCE_PEAK`, `VAD_HANGOVER_MS`, `RECOG_PATIENCE_MS`. (In v2 these move into the `Role` — `recog_patience` is a Role field per ARCHITECTURE §Data model; here it is firmware config.)
- Integrate into `recordWhileHeld` so capture can stop on a detected pause, not only on release; keep the v1.3 noise/length gate (`shouldTranscribe`) afterward.
- Tune the push-to-talk ↔ pause-detection interplay: release always ends immediately; a pause ends after the hangover; barge-in (interrupting playback) is **out of scope** this phase.

**Dependencies:** PYR-014 (uses the `Listening` state).

**Expected result:**
The user speaks a phrase and the device detects the end by the natural trailing pause, without holding/releasing precisely; holding the button still works.

**Acceptance criteria:**
- [ ] Endpointing helper is pure + host-tested (silence→end, patience cap, no false trigger on quiet, no premature cut on short gaps)
- [ ] `recog_patience`, silence threshold, and hangover loaded from `config.h`
- [ ] Speaking with a trailing pause ends capture and triggers the turn (no button release needed)
- [ ] Push-to-talk (hold/release) still works and feels natural alongside pause detection
- [ ] Verified on hardware across short/long utterances and varied pauses

### PYR-016 — Mid-turn resilience (Wi-Fi loss + per-stage timeouts)

**Description:**
Harden the turn against failures partway through: Wi-Fi dropping mid-turn, or an ASR/LLM/TTS stage stalling. The loop must surface an **Error**/**Offline** state and recover to **Idle** cleanly — never hang or get stuck on a stale state, and never leave the shared mic/speaker bus half-open.

**What needs to be done:**
- Audit the per-stage timeout budgets (ASR/LLM/TTS — most already exist as `kAsrReadMs`/`kHttpReadMs`/`kTtsReadMs`) so each stage aborts within a bounded time → `Error` → back to `Idle`.
- Detect Wi-Fi loss mid-turn (the v0.3 `serviceWiFi` already tracks `g_offline`): abort the in-flight turn, show `Offline`, pause input, and resume `Idle` on reconnect.
- Guarantee the mic↔speaker bus is left in a clean state (mic mode) on **any** mid-turn abort — no half-open speaker/mic after a failure.
- Surface a readable error on serial + the LCD `Error` state, then auto-return to `Idle` after a short dwell.
- Note: the **enumerated `error.code` wire set** (ARCHITECTURE §Error handling) is a **v2.1** concern (it rides the WS contract); in v1.4 errors are local LCD + serial only.

**Dependencies:** PYR-014 (the `Error`/`Offline` states).

**Expected result:**
Any mid-turn failure (Wi-Fi drop, stage timeout) ends in a clean `Idle` with a clear message; the device never hangs and the audio bus is always recovered.

**Acceptance criteria:**
- [ ] Each stage (ASR/LLM/TTS) has a bounded timeout; on timeout the turn aborts to `Error` → `Idle`
- [ ] Wi-Fi loss mid-turn aborts cleanly, shows `Offline`, and recovers to `Idle` on reconnect
- [ ] The audio bus (mic/speaker) is always restored to mic mode after an abort
- [ ] Errors shown on LCD + serial; loop never hangs (verified with induced failures: pulled Wi-Fi, forced stage stall)

### PYR-017 — Latency hardening (pre-warm ASR TLS + timing attribution)

**Description:**
Cut the felt latency the v1.3 instrumentation exposed — ASR was **~61 %** of the post-speech wait, dominated by **~3 s of per-call connection overhead** (`ASR ≈ 3043 ms + 1.14 × clip_ms`, r ≈ 0.75) — and fix the instrumentation gap so the breakdown stays trustworthy as latency work proceeds. Carries over recommendations **#2** and **#5** from the v1.3 latency analysis.

**What needs to be done:**
- **Pre-warm the ASR connection (#2):** start the TLS connection to the ASR host when capture begins (button-press / `Listening` entry) so the handshake overlaps the user's speech instead of being paid after release on the critical path. Reuse the warmed connection for the POST; fall back to a fresh connect if it dropped or wasn't ready.
- **Fix timing attribution (#5):** stamp `g_stamps.asrMs` immediately after `asrTranscribe` regardless of outcome (success / empty / low-confidence / error) — today it is only stamped on success, so re-prompt turns mis-report `asr=0` and dump the time into `other` (observed: a re-prompt turn logged `asr=0 ... other=4982`).
- **Measure:** compare the ASR stage's fixed overhead and `press→speak` before/after on hardware; record the delta in the execution report.

**Dependencies:** PYR-014 (pre-warm fires on the `Listening` entry).

**Expected result:**
The ASR stage's fixed overhead drops measurably, and the `[latency]` breakdown is correct on every turn — including re-prompts.

**Acceptance criteria:**
- [ ] The ASR TLS connection is pre-warmed at capture start and reused for the POST (fresh-connect fallback if dropped)
- [ ] `asrMs` is attributed on every outcome (success / empty / low-conf / error); re-prompt turns no longer show `asr=0`
- [ ] `timing.h` host tests still pass; a case covers the re-prompt attribution path (`pio test -e native`)
- [ ] Hardware measurement shows reduced ASR fixed overhead vs the v1.3 baseline (numbers recorded)

## v1.4 scope notes

**Total effort:** ~1.5 weeks for a single developer. PYR-014 unblocks the other three, which then run in parallel; PYR-015 (VAD) carries the bulk.

**Critical path:** PYR-014 → PYR-015. The "state always visible" half of the DoD is met at PYR-014; reliability (PYR-015/016) and responsiveness (PYR-017) complete it.

**Phase DoD (ROADMAP §v1.4):** the voice chat works reliably, the current state is always visible on the screen, the `[latency]` breakdown is correct on every turn (including re-prompts), and pre-warming measurably cuts the ASR stage's fixed overhead versus the v1.3 baseline.

**Still direct-to-cloud:** v1.4 closes out v1 without an own server. The orchestration stays on the device; v2.1 moves the ASR→LLM→TTS loop server-side (and is where streaming ASR / sentence-streaming TTS / early playback — recs #1/#3/#4 — land).

**Config vs. Role:** the new `recog_patience` / VAD knobs live in firmware `config.h` for v1; in v2.2 they become `Role` fields (config is the source of truth, just relocated server-side). No behavior is hardcoded that belongs to the role.

**Reused infrastructure:** the v1.1 capture + `analyzePcm`/`shouldTranscribe` (PYR-006/013) for VAD levels, the v0.3 Wi-Fi supervisor + `backoff.h` for resilience, and the v1.3 `timing.h` / ASR client for the latency work. v1.4 adds the state machine and the VAD/endpointing helper as new pure headers.
