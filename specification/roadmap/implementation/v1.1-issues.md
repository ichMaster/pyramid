## v1.1 — Audio I/O and PlatformIO migration

### PYR-004 — PlatformIO migration

**Description:**
Move the firmware off the Arduino IDE onto **PlatformIO** and establish the project layout reused for the rest of v1–v3. Behavior must not change — this is a build-system migration of the working v0 firmware, not a feature change.

**What needs to be done:**
- Add `firmware/platformio.ini`: define the **AtomS3R** environment (ESP32-S3 + `arduino` framework — select an existing board id or supply a custom board JSON if AtomS3R isn't built in), `lib_deps` for **M5Unified** and **ArduinoJson** (pin versions), `monitor_speed = 115200`, and partition/flash settings matching the current build.
- Reorganize sources into the PlatformIO layout: move the sketch into `firmware/src/` (`pyramid.ino` → `src/main.cpp`, or keep the `.ino` under the Arduino framework) and place the pure headers alongside (`src/` or `include/`). Keep the `config.example.h` → `config.h` flow intact; `config.h` stays gitignored.
- Confirm `pio run` builds and `pio run -t upload` flashes, producing the same behavior as the Arduino build (boot, Wi-Fi, serial chat, streaming reply).
- Update `firmware/README.md`: PlatformIO build/flash instructions, and a one-line note of where the `.ino` moved.

**Dependencies:** None (first v1 issue; builds on the existing v0 firmware).

**Expected result:**
The existing v0 firmware builds and flashes from PlatformIO with identical behavior; sources live under `firmware/src/`.

**Acceptance criteria:**
- [ ] `firmware/platformio.ini` defines the AtomS3R env (ESP32-S3 + Arduino framework; M5Unified + ArduinoJson in `lib_deps`)
- [ ] `pio run` compiles the firmware cleanly
- [ ] `pio run -t upload` flashes the board (manual, on hardware)
- [ ] Sources moved into the PlatformIO layout; `config.h` still gitignored and not committed
- [ ] Runtime behavior matches the v0 Arduino build (boot → Wi-Fi → serial chat) — manual on-device check
- [ ] `firmware/README.md` documents the PlatformIO build/flash and where the `.ino` moved

### PYR-005 — Native test environment

**Description:**
Add PlatformIO's **native (host) test environment** so the pure v0 logic gains formal, CI-ready automated coverage — fulfilling the promise carried since v0 ("host-testable logic folds into the native env in v1").

**What needs to be done:**
- Add a `native` environment to `platformio.ini` (`platform = native`) and arrange `test/` so `pio test -e native` discovers the suites.
- Port the existing host tests — `line_reader`, `chat_api`, `history`, `backoff`, `sse` — into the native env (adapt to Unity, PlatformIO's default test framework, or run the existing assert-style mains under native). Resolve the **ArduinoJson** include for the native build (`lib_deps` / include path).
- Ensure the pure headers (`line_reader.h`, `serial_protocol.h`, `chat_api.h`, `history.h`, `backoff.h`, `sse.h`) compile under native with no Arduino dependency.
- Update `firmware/README.md` to document `pio test -e native` (replacing the manual `c++` invocations), and prepare for CI wiring (`.github/workflows/ci.yml`) per ARCHITECTURE §Testing and CI.

**Dependencies:** PYR-004

**Expected result:**
`pio test -e native` runs all pure-logic suites green on the host, ready to gate CI.

**Acceptance criteria:**
- [ ] `native` env in `platformio.ini`; `pio test -e native` discovers and runs the suites
- [ ] All ported tests pass (line reader, chat api, history, backoff, sse) — matching the current host-test coverage
- [ ] ArduinoJson resolves for the native build
- [ ] Pure headers compile under native without Arduino deps
- [ ] `firmware/README.md` documents `pio test -e native` as the host-test entry point

### PYR-006 — I2S audio capture (push-to-talk)

**Description:**
Bring up the **Echo Base microphone** over I2S and record **16 kHz mono PCM16** while the button is held (push-to-talk), with basic input safety. This is the capture half of the v1.1 hardware foundation.

**What needs to be done:**
- Initialize mic input (via M5Unified's `Mic` API or the ESP32 I2S driver), **16 kHz mono, PCM16**.
- Record into a fixed PCM16 buffer while `BtnA` is held; stop on release.
- Add basic **level/clipping checks** (peak/RMS, clip flag) and a **fixed maximum record duration** that caps the buffer; log captured length/level to serial (gated by `debug_serial`).
- Keep buffer/duration/level math in pure helpers where feasible so it is host-testable under `pio test -e native`.

**Dependencies:** PYR-004

**Expected result:**
Holding the button records mic audio into a PCM16 buffer at 16 kHz mono, bounded by a max duration, with level/clip reporting on serial.

**Acceptance criteria:**
- [ ] Mic captures 16 kHz mono PCM16 into a buffer while the button is held
- [ ] A fixed maximum record duration caps the buffer; recording stops on release or at the cap
- [ ] Basic level/clipping check is computed and logged (gated by `debug_serial`)
- [ ] Captured length/level is visible on serial
- [ ] Buffer/duration/level helpers covered by a native host test where feasible; on-device capture is a manual check

### PYR-007 — I2S audio playback (record → playback loop)

**Description:**
Bring up the **Echo Base speaker** over I2S and play audio on button release — completing the push-to-talk **record → playback** loop that is v1.1's DoD.

**What needs to be done:**
- Initialize speaker output (M5Unified's `Speaker` API or the ESP32 I2S driver), matching the capture format (**16 kHz mono PCM16**).
- On button release, play back through the speaker — the **just-captured buffer** (loopback, directly demonstrating the DoD), with a small bundled test clip as a fallback.
- Handle playback start/stop cleanly with a basic volume; don't block the main loop longer than the clip; return to idle.
- Coordinate with capture (PYR-006): hold records, release plays.

**Dependencies:** PYR-004, PYR-006

**Expected result:**
Releasing the button plays audio through the Echo Base speaker; together with capture, the device records on hold and plays it back on release.

**Acceptance criteria:**
- [ ] Speaker plays 16 kHz mono PCM16 audio on button release
- [ ] The just-captured buffer (or a test clip) is audible through the Echo Base
- [ ] Press-records / release-plays-back works end to end (with PYR-006) — manual on-device check
- [ ] Playback does not hang the loop; the device returns to idle cleanly

## v1.1 scope notes

**Total effort:** ~1–1.5 weeks for a single developer (PYR-004 gates the rest; PYR-005/006/007 then largely parallel).

**Critical path:** PYR-004 → PYR-006 → PYR-007 (the record→playback loop that proves the phase). PYR-005 is parallel and gates CI rather than the DoD.

**Phase DoD (ROADMAP §v1.1):** the project builds and flashes from PlatformIO, and press-records / release-plays-back works through the Echo Base.

**Firmware toolchain note:** v0 was built in the Arduino IDE; from v1.1 the firmware is **PlatformIO** (`pio run`, `pio test -e native`). The pure v0 headers carry over unchanged and are now formally tested in the native env.

**Hardware validation:** I2S capture/playback, the mic, and the speaker require the AtomS3R + Echo Base; those acceptance items are manual on-device checks. Host-testable logic (buffer/duration/level math, the ported v0 parsers) is covered by `pio test -e native`.
